# 内存安全

Toka 通过 **PAL Checker（Pointer Analysis Layer，指针分析层）**在编译时实现内存安全，无需垃圾回收器、运行时开销或显式生命周期标注。

## PAL 如何工作

PAL Checker 分析代码中帽子标记符号（`*`、`^`、`~`）的使用情况，强制执行所有权和借用规则。它在编译时运行并保证：

1. **无悬垂指针** — 在内存释放后不能访问
2. **无双重释放** — 每次分配恰好释放一次
3. **移动后不可使用** — 被移动的 `^`（独占）指针不能访问
4. **无数据竞争** — 可变访问是独占的；共享访问是只读的

## 独占指针（`^`）— 移动后不可使用

`^` 帽子强制执行独占所有权。所有权可以通过*移动*转移：

```toka
{{#include ../../examples/memory_safety.tk:no_use_after_move}}
```

编译器追踪所有权的流向，并拒绝任何在所有权转移后访问值的代码。

## 原始指针（`*`）— 不安全操作

原始指针需要显式的 `unsafe` 块：

```toka
{{#include ../../examples/memory_safety.tk:unsafe_ops}}
```

`unsafe` 关键字向 PAL Checker 发出信号，表示你在手动承担内存安全的责任。

## 共享指针（`~`）— 引用计数

共享指针使用运行时引用计数，并且是完全安全的。它们允许多个指针共享对同一堆分配资源的所有权。

```toka
auto ~s1# = new Point(x = 1000, y = 2000, z = 0)
{
    auto ~s2# = ~s1 // 共享拷贝（引用计数增加）
    s2.x = 3000     // 直接修改底层数据（灵魂）
} // 当 s2 离开作用域时，引用计数减少
```

## 借用安全

在 Toka 中，函数参数默认是不可变的，并且是按值传递（拷贝）的。无论是简单类型（如 `i32`）还是复杂类型（如自定义 `shape` 类型），在传递给函数时都会执行拷贝（结构体执行浅拷贝），从而保护调用方的参数不被修改：

```toka
{{#include ../../examples/memory_safety.tk:borrowing_safe}}
```

对于**可变访问**，在局部变量声明上使用 `#`：

```toka

```

## PAL Checker 的保证一览

| 情况 | PAL Checker 行为 |
|------|-----------------|
| `^` 移动后使用 | ❌ 编译错误 |
| 悬垂的 `&` 引用 | ❌ 编译错误 |
| 双重 `unsafe free` | ❌ 编译错误 |
| 可变数据竞争 | ❌ 编译错误 |
| 有效的借用使用 | ✅ 通过 |
| 正确的作用域清理 | ✅ 自动释放 |

该模型实现了与 Rust 借用检查器相同的安全保证，但无需任何生命周期标注——帽子标记符号直接在语法中使所有权变得可见。
